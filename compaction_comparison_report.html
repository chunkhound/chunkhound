Compaction Strategy Comparison

ChunkHound native (stack/1-compaction) vs. chunkhound-index-compactor (community) | 2026-05-25

Root Cause — What Bloat Actually Is

Both tools solve the same fundamental problem: DuckDB databases accumulate storage waste that VACUUM cannot reclaim. But they target different manifestations of the same underlying issue.

Their benchmark: 1.25 TB bloated index containing only ~2 GiB of live data — a 600:1 bloat ratio — caused by ChunkHound's incremental indexing workflow.
SolutionRoot Cause Targeted
Ours Partially-deleted row groups + block fragmentation from DROP INDEX cycles. DuckDB CHECKPOINT frees only entirely-empty row groups — partially-deleted groups stay allocated.
Theirs ChunkHound's incremental write batch triggers DROP INDEX + recreate when batches exceed 50 rows. Old HNSW serializations are orphaned in the DuckDB file. VSS re-serializes the entire HNSW on every CHECKPOINT, compounding the churn.

Both mechanisms are real and co-occur in production. Their tool targets the dominant mechanism for ChunkHound's specific write workload.

How Each Mechanism Works

Our Approach — EXPORT / IMPORT / Swap

1. Measure: max(free_block_ratio, row_waste_ratio) >= threshold?
2. EXPORT DATABASE → Parquet files  (disk: +1x original size)
3. IMPORT DATABASE → new .duckdb    (disk: +1x again → peak 3x)
4. Atomic swap: original → .old, compact → original
5. Delete .old, reconnect
6. Force full reindex (mtime unreliable post-swap)
Peak cost: 3× disk space. Minimal RAM. HNSW dropped — rebuilt on next reindex pass.

Their Approach — In-Memory Topological Copy

1. Open in-memory DuckDB (temp_directory to avoid filling disk)
2. Attach source DB read-only
3. Extract schema DDL; separate HNSW index definitions
4. Topological sort tables by FK dependencies
5. Stream INSERT parent-before-child (avoids FK race bugs)
6. Recreate non-HNSW indexes verbatim
7. Recover HNSW params via pragma_hnsw_index_info()
8. Rebuild HNSW in destination (OR write recipe table + defer via --skip-hnsw)
9. CHECKPOINT, detach, replace original
Peak cost: 3–4× HNSW size in RAM. Low disk. HNSW preserved and immediately usable.
Why they can't use DuckDB's native COPY FROM DATABASE

DuckDB's native COPY FROM DATABASE has a foreign-key insertion-order race condition that causes unpredictable failures at scale — the violating key differs between runs even with referentially clean data. Their topological sort explicitly resolves this. Our EXPORT/IMPORT handles it transparently because Parquet files carry no FK constraints.

Why HNSW metric recovery is non-trivial

DuckDB catalog stores HNSW DDL, but strips the WITH (...) clause that contains metric='cosine'. Their tool uses pragma_hnsw_index_info() to recover the metric before recreating. Without this, silently defaults to L2 distance — wrong results, no error.

P2 — Post-ship

Borrow their topological sort

EXPORT/IMPORT handles FK ordering transparently today. But if we ever add a streaming-copy fallback or support partial compaction, their topological sort avoids the DuckDB COPY FROM DATABASE race bug. Worth porting as a utility.

P2 — Post-ship

Write migration guide

Users currently running chunkhound-index-compactor manually need a clear path to our native solution. Document: how to detect if you need it, how to disable the external tool, and how to verify automatic compaction is running.

P3 — Nice-to-have

Expose standalone repack

Our chunkhound repack currently requires ChunkHound context. Consider making it usable as a recovery tool when the MCP server won't start — useful when the DB itself is the bloat problem causing startup timeout.

P3 — Nice-to-have

Ship as complement, not replacement

The two tools are not mutually exclusive. Our solution handles day-to-day automatic maintenance. Their tool handles emergency TB-scale recovery where HNSW must survive intact. Document this positioning clearly.

Do not block the PR on the scale benchmark alone — but do document the known limitation and create a follow-up issue to validate at scale before recommending ChunkHound for TB-scale deployments.
