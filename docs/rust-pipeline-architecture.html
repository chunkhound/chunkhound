2. Design Principles

  1. Streaming pipeline — embed then store. The main loop is a 3-thread pipeline: parse → embed → store. Each batch flows through dedicated threads for CPU (parse), network (embed), and disk (store). All three threads run concurrently on different batches — bounded channels provide natural backpressure.
  2. One DB write per chunk. Because embedding runs before storage, chunks and their embedding vectors are inserted together in a single DuckDB transaction. No separate embedding storage pass. No embed_cache to manage.
  3. Parsers and embedding providers stay in Python. These are complex subsystems that use external SDKs (tree-sitter, openai, httpx) and evolve independently. Rust treats them as opaque callbacks with typed contracts.
  4. GIL is not a bottleneck. Parse threads hold the GIL for ~100µs per file. Embed thread holds the GIL only during argument marshalling — httpx releases it during HTTP calls. Store thread never touches the GIL. All three threads make concurrent progress.
  5. No json.dumps round-trip. Data crosses the Rust/Python boundary as typed structs via PyO3 extraction — the same zero-copy pattern RustDbWriter already uses.
  6. Backward compatible. The Python IndexingCoordinator and DirectoryIndexingService remain available. The Rust pipeline is an alternative entry point, configurable via a toggle.
  7. Progressive migration. Each phase can be toggled independently via feature flags or config. Fall back to the Python path on error. Build → test → benchmark each phase before proceeding.

4. Rust Implementation — Main Flow

4.1 IndexingPipeline — The Orchestrator New

The main PyO3 class. Python calls .run() from asyncio.to_thread. Internally, run() spawns three threads connected by bounded channels.

Thread safety requirement: The parse thread uses rayon::par_iter and the embed thread uses thread::spawn — both call Python callbacks from non-main threads. The .so must call pyo3::prepare_freethreaded_python() in the module init function (#[pymodule_init]). This is a new requirement — the existing codebase does not call it yet.
The entire module is gated behind #[cfg(feature = "rust-pipeline")] — the .so only includes this code when the feature is enabled.

/// src/pipeline/mod.rs — entire module is feature-gated

/// src/pipeline/pipeline.rs
#[cfg(feature = "rust-pipeline")]
#[pyclass]
pub struct IndexingPipeline {
    config: PipelineConfig,
    state: PipelineState,
}

#[pymethods]
impl IndexingPipeline {
    #[new]
    fn new(config: &Bound<PyDict>) -> PyResult<Self>;

    /// Run the full indexing pipeline synchronously.
    /// Call from Python via `asyncio.to_thread(pipeline.run, ...)`.
    fn run(
        &mut self,
        py: Python<'_>,
        files: Vec<String>,                      // discovered by Python, passed to Rust
        parse_callback: Py<PyAny>,              // (path, detect_sql) → (language, list[dict])
        embed_callback: Option<Py<PyAny>>,       // (texts, provider, model) → list[list[float]]
        progress_callback: Option<Py<PyAny>>,     // (phase, cur, total, info) → bool
    ) -> PyResult<PipelineReport>;
}
Note: embed_cache is gone. Texts flow from parse → embed thread through an mpsc::sync_channel. Nothing accumulates across batches except the DB snapshot (a HashMap that shrinks as files are matched).

/// Produced by the parse thread, consumed by the embed thread.
pub struct DiffResult {
    // ── file metadata (for store thread to upsert files table) ──
    pub file_id: i64,                     // 0 if new file (store thread queries by path)
    pub path: String,                     // relative path (DB key)
    pub name: String,                     // filename only (NOT NULL in DB)
    pub extension: Option<String>,         // file extension
    pub language: Option<String>,
    pub file_size: u64,
    pub mtime: f64,
    pub content_hash: String,
    pub is_new_file: bool,

    // ── chunks (embed_text + embedding fields filled downstream) ──
    pub chunks_to_insert: Vec<NewChunk>,
}

/// Produced by the embed thread, consumed by the store thread.
/// Embeddings are already populated on NewChunk.embedding — no separate vectors list.
pub struct EmbeddedBatch {
    pub diffs: Vec<DiffResult>,           // NewChunk.embedding is Some for embeddable chunks
}

/// One bulk SELECT at startup — zero per-file DB queries during diff.

4.3 Main Loop — Three-Thread Streaming

use std::sync::mpsc;
use std::thread;

fn run(&mut self, py, files,
       parse_cb, embed_cb, progress_cb) -> PipelineReport {

    let started = Instant::now();

    // ═══ Phase A: Setup ════════════════════════════════════
    let db_path = self.config.db_path.clone();
    py.allow_threads(|| {
        let db = RustDbWriter::open(&db_path)?;
        db.drop_hnsw_indexes()?;
    });
    let mut snapshot = py.allow_threads(|| load_db_snapshot(&db_path))?;

    // Files discovered by Python — passed as argument to run()
    let total_files = self.files.len();
    let mut files_iter = self.files.into_iter().peekable();

    // ═══ Channels ══════════════════════════════════════════
    // Bounded capacity 2: backpressure prevents memory blow-up.
    let (parse_tx, parse_rx) = mpsc::sync_channel::<Vec<DiffResult>>(2);
    let (store_tx, store_rx) = mpsc::sync_channel::<EmbeddedBatch>(2);

    // ═══ Embed thread — attaches vectors directly to chunks ═══
    let embed_config = self.config.clone();
    let embed_cb = embed_cb.clone();
    let embed_handle = thread::spawn(move || {
        for mut diffs in parse_rx {
            // Collect texts from chunks that need embedding
            let texts: Vec<&str> = diffs.iter()
                .flat_map(|d| d.chunks_to_insert.iter())
                .filter(|c| c.embed_text.is_some())
                .map(|c| c.embed_text.as_ref().unwrap().as_str())
                .collect();

            if !texts.is_empty() && !embed_config.skip_embeddings && embed_cb.is_some() {
                // Call Python embed_callback — catch errors, store chunks without vectors
                let vectors_result = Python::with_gil(|py| {
                    let cb = embed_cb.as_ref().unwrap().bind(py);
                    cb.call1((
                        texts,
                        embed_config.embedding_provider.as_str(),
                        embed_config.embedding_model.as_str(),
                    ))
                    .and_then(|r| r.extract::>>(py))
                });

                if let Ok(vectors) = vectors_result {
                    // Validate: provider must return one vector per text
                    if vectors.len() != texts.len() {
                        // Mismatch — skip embedding, store chunks bare
                        // (no panic — pipeline continues)
                    } else {
                        // Attach vectors back to chunks (same iteration order)
                    let mut vi = 0;
                    for diff in diffs.iter_mut() {
                        for chunk in diff.chunks_to_insert.iter_mut() {
                            if chunk.embed_text.is_some() {
                                chunk.embedding = Some(vectors[vi].clone());
                                chunk.provider = Some(embed_config.embedding_provider.clone());
                                chunk.model = Some(embed_config.embedding_model.clone());
                                vi += 1;
                            }
                        }
                    }
                }
                // On error: chunks stay with embedding=None — stored bare, user can re-embed
            }
            // Send populated chunks to store thread
            store_tx.send(EmbeddedBatch { diffs }).unwrap();
        }
    });

    // ═══ Store thread — builds DbWriterBatch, calls existing backend ═══
    let store_config = self.config.clone();
    let store_handle = thread::spawn(move || {
        let mut db = RustDbWriter::open(&store_config.db_path)
            .expect("store thread: failed to open DB");
        let mut total_stored = 0u64;
        for batch in store_rx {
            // Build a DbWriterBatch from DiffResults — reuses existing
            // DuckDbHnswBackend::write_batch() with temp tables, FK workarounds,
            // 100-row batching, and embeddings_N table sharding.
            let writer_batch = build_writer_batch(&batch.diffs);
            total_stored += db.write_batch(writer_batch)
                .expect("store thread: batch write failed");
        }
        // Explicit close to CHECKPOINT before Phase C opens another connection
        db.close().expect("store thread: close failed");
        total_stored
    });

    // ═══ Parse thread (main thread) ═══════════════════════
    while files_iter.peek().is_some() {
        let batch_paths: Vec = files_iter
            .by_ref()
            .take(self.config.parse_batch_size)
            .map(PathBuf::from)
            .collect();
        let parsed = self.parse_batch(py, &batch_paths, &parse_cb);

        // Separate errored files from successful ones
        let (ok_files, err_files): (Vec<_>, Vec<_>) =
            parsed.into_iter().partition(|f| f.error.is_none());
        self.state.errors.extend(
            err_files.iter().filter_map(|f| f.error.clone())
        );

        if !ok_files.is_empty() {
            let diffs = self.diff_and_remove(&ok_files, &mut snapshot)?;
            self.state.record_parsed(&ok_files);
            self.emit_progress(py, &progress_cb, total_files)?;

            // Release GIL before blocking send (embed thread is GIL-free)
            py.allow_threads(|| parse_tx.send(diffs).expect("parse_tx: embed thread dropped"));
        }
    }
    drop(parse_tx);  // signals embed thread to finish

    // ═══ Drain ════════════════════════════════════════════
    embed_handle.join().expect("embed thread panicked");
    let chunks_written = store_handle.join().expect("store thread panicked");

    // Phase C: Finalize
    py.allow_threads(|| {
        let db = RustDbWriter::open(&db_path)?;
        db.delete_orphaned_files(&snapshot)?;
        db.run_compaction()?;
        db.rebuild_hnsw_indexes()?;
        db.close()?;
    });

    self.state.chunks_written = chunks_written;
    Ok(self.state.into_report(started.elapsed()))
}
Key simplification: embed-before-store eliminates embed_cache, store_embeddings(), the two-phase ID tracking, and the separate embedding storage pass. The store thread does one INSERT per chunk (chunk + embedding together). The embed thread attaches vectors directly to NewChunk fields — no fragile parallel-vector mapping. When skip_embeddings is true (or embed_callback is None), the embed thread passes batches straight through with no HTTP calls.

/// Build a DbWriterBatch from DiffResults — delegates to existing
/// DuckDbHnswBackend::write_batch() with temp tables and embeddings_N sharding.
fn build_writer_batch(diffs: &[DiffResult]) -> DbWriterBatch {
    let mut batch = DbWriterBatch::new();

    for diff in diffs {
        // File record (for upsert_file in backend)
        batch.files.push(FileRecord {
            id: if diff.is_new_file { None } else { Some(diff.file_id) },
            path: diff.path.clone(),
            name: diff.name.clone(),
            extension: diff.extension.clone(),
            size_bytes: Some(diff.file_size as i64),
            mtime: Some(diff.mtime),
            content_hash: Some(diff.content_hash.clone()),
            language: diff.language.clone(),
        });

        // Chunks to delete for modified files
        if !diff.is_new_file {
            batch.chunk_ids_to_delete.push(diff.file_id);  // DELETE BY file_id
        }

        // Chunks + embeddings to insert
        for chunk in &diff.chunks_to_insert {
            batch.chunks.push(ChunkRecord {
                id: None,  // auto-incremented by DB
                file_id: 0, // resolved by backend after file upsert
                chunk_type: chunk.chunk_type.clone(),
                symbol: chunk.symbol.clone(),
                code: chunk.code.clone(),
                start_line: chunk.start_line,
                end_line: chunk.end_line,
                start_byte: chunk.start_byte,
                end_byte: chunk.end_byte,
                language: chunk.language.clone(),
                metadata: chunk.metadata.clone(),
                embedding: chunk.embedding.clone(),
                provider: chunk.provider.clone(),
                model: chunk.model.clone(),
            });
        }
    }

    batch
}
Why reuse the existing backend: DuckDbHnswBackend::write_batch() already handles temp-table staging, FK pre-deletion, embeddings_{dims} table sharding, 100-row batching, and HNSW lifecycle. Building a DbWriterBatch from DiffResults and delegating avoids reinventing SQL, embedding JSON marshalling, and connection management.
Implementation note — pseudocode vs actual types: The pseudocode in this document illustrates logical flow. The implementation must match the existing codebase types:
  • FileRecord uses existing_file_id: Option<i64> (not id). name/extension are derived from path internally by upsert_file().
  • DbWriterBatch nests chunks as FileRecord.chunks: Vec<ChunkRecord>. Per-file deletion needs a new delete_file_ids: Vec<i64> field.
  • RustDbWriter is a PyO3 class — its methods require Python<'_>. Store thread must use Python::with_gil(|py| ...). Pipeline should call DbBackend::write_batch() directly (bypassing the Python dict wrapper).
  • pyo3::prepare_freethreaded_python() must be called in module init for the embed thread.
  • Parse callback returns PyAny — use .extract::<(String, &PyList)>(). Method names: drop_all_hnsw_indexes(), ensure_all_hnsw_indexes(). write_batch() returns BatchResult.
These details are resolved during implementation by reading the existing source. The architecture describes design — the pseudocode illustrates intent.

4.5 Parse Concurrency — Rayon Within Each Batch

Parsing is CPU-bound (tree-sitter C library). Within each batch, files are parsed in parallel via rayon. This sits inside Thread 1 — the parse thread uses rayon internally.

fn parse_batch(
    &self, py: Python,
    files: &[PathBuf],
    callback: &Py<PyAny>,
) -> Vec<ParsedFile> {  // errors captured in ParsedFile.error

    py.allow_threads(|| {
        files.par_iter()
            .map(|path| {
                Python::with_gil(|py| {
                    let cb = callback.bind(py);
                    // Callback returns (language: str, chunks: list[dict])
                    // Language detection is done in Python (40+ mappings)
                    let result = cb.call1((
                        path.to_string_lossy().as_ref(),
                        self.config.detect_embedded_sql,
                    ));
                    match result {
                        Ok((lang_py, chunks_py)) => {
                            let lang: String = lang_py.extract(py).unwrap_or_default();
                            extract_parsed_result(py, path, &lang, chunks_py)
                        },
                        Err(e) => ParsedFile {
                            path: path.clone(),
                            language: None,
                            file_size: 0, mtime: 0.0, content_hash: String::new(),
                            chunks: Vec::new(),
                            error: Some(e.to_string()),
                        },
                    }
                })
            })
            .collect()
    })
}
Why threads, not processes: tree-sitter's C library releases the GIL internally during parsing. Each rayon thread holds the GIL for ~100µs per file (the thin Python adapter). No fork overhead, no pickling, shared tree-sitter grammars. Same create_parser_for_language + parser.parse_file() — just called from a shared process.

4.6 Garbage Collection

Deletions happen in Thread 3 (Store) per-batch, and once after all batches for orphaned files:

CaseDetected byWhat happens
Modified file diff_and_remove(): content hash changed → diff produced with new chunks Store thread: DELETE FROM chunks WHERE file_id = ? (all old chunks, even unchanged ones — they get re-inserted). Then INSERT new chunks with attached embeddings.
Deleted file After parse thread finishes, remaining snapshot entries were never matched Finalize: DELETE FROM files WHERE id IN (...) — chunks + embeddings cascade
Unchanged file mtime+size unchanged, content hash matches → removed from snapshot, no diff produced Nothing — skipped entirely

4.7 Reindexing Flow

Reindexing is just calling run() again on an already-indexed directory. The snapshot+diff mechanism handles it — unchanged files are completely skipped.

Reindexing a directory with existing DB: ═══ Phase A: Load DB snapshot ═══════════════════════ SELECT path, id, size, modified_time, content_hash FROM files → HashMap<String, DbFileState> ┌────────────────────────────────────────────────────┐ │ DB has: main.rs (id=1, hash=abc) │ │ lib.rs (id=2, hash=def) │ │ old.rs (id=3, hash=ghi) ──┐ │ └─────────────────────────────────────────────│──────┘ │ ═══ Phase B: Parse + Diff (per file on disk) ═│══════ │ Files on disk: │ │ main.rs ── parse → mtime+size unchanged │ hash still "abc" → SKIPPED │ removed from snapshot ✓ │ │ lib.rs ── parse → mtime changed │ hash now "xyz" ≠ "def" → MODIFIED │ DiffResult { file_id=2, │ is_new_file=false, │ chunks_to_insert=[...] } │ → flows through channels │ → store: DELETE old chunks, │ INSERT new chunks + embeddings │ │ new.rs ── parse → not in snapshot → NEW │ DiffResult { file_id=0, │ is_new_file=true, │ chunks_to_insert=[...] } │ → store: INSERT file + chunks │ │ ═══ After all files processed ════════════════│══ │ Snapshot remaining: │ old.rs (id=3) ← never matched ────────────┘ → orphaned_file_ids = [3] → DELETE FROM files WHERE id=3 (cascades: DELETE chunks, DELETE embeddings) ═══ Phase D: Finalize ═══════════════════════════════ compact → rebuild HNSW → close Result: 1 file skipped, 1 file updated, 1 file added, 1 file deleted.
Key property: unchanged files never enter the channels, never touch the DB, and don't consume parse/embed/store time beyond the initial snapshot lookup. For a repo where 90% of files are unchanged, reindexing processes only the 10% that changed.

5.4 Pipeline Bridge Module (chunkhound/pipeline_bridge.py)

# chunkhound/pipeline_bridge.py

from pathlib import Path
from chunkhound.parsers.parser_factory import create_parser_for_language
from chunkhound.core.types.common import Language, FileId

def parse_file_callback(
    file_path: str,
    detect_embedded_sql: bool = True,
) -> tuple[str, list[dict]]:
    """Adapter: path → (language, chunks).

    All parse-tuning flags are handled internally via global config:
    - per_file_timeout_seconds / per_file_timeout_min_size_kb
    - config_file_size_threshold_kb (skip oversized config files)
    - max_file_size_mb (skip files over the limit)
    - index_unknown_files (return empty for unrecognized extensions)

    Rust passes only detect_embedded_sql — everything else is baked in.
    """
    from chunkhound.registry import get_registry
    from chunkhound.services.batch_processor import detect_language

    # Skip files exceeding max_file_size_mb
    config = get_registry().indexing_config
    if Path(file_path).stat().st_size > config.max_file_size_mb * 1024 * 1024:
        return ("", [])
    lang = detect_language(file_path)
    if lang is None:
        return ("", [])
    parser = create_parser_for_language(lang,
                detect_embedded_sql=detect_embedded_sql)
    if parser is None:
        return (lang.value, [])
    file_id = FileId(0)  # Rust assigns the real ID
    chunks = parser.parse_file(Path(file_path), file_id)
    return (lang.value, [c.to_dict() for c in chunks])


def embed_callback(
    texts: list[str],
    provider: str,
    model: str,
) -> list[list[float]]:
    """Adapter: batch embed (called from Rust embed thread).

    All embedding config is baked into the service via global config —
    api_key, base_url, ssl_verify, output_dims, client_side_truncation,
    batch_size, timeout, max_retries, max_concurrent_batches, Azure config.
    """
    from chunkhound.registry import get_registry
    service = get_registry().embedding_service
    return service.generate_embeddings(texts, provider, model)

10. Key Design Decisions

Why embed
before store?		 One DB write per chunk (chunk + embedding together). No embed_cache, no two-phase ID tracking, no separate embedding storage pass. Embeddings are attached directly to NewChunk.embedding by the embed thread — no fragile parallel-vector mapping. The embed thread is a no-op pass-through for --skip-embeddings.
Three threads
not just two?		 Parse (CPU), embed (network), and store (disk) use different resources. Running all three concurrently on different batches hides the full embed and store time behind parse. Two threads (parse+store together, embed separately) would serialize parse and store — but store is only ~30ms per batch vs parse's ~600ms, so that marginal gain is small. Three threads cost very little additional complexity (two channels instead of one) and give the cleanest separation of concerns.
Why channels
with capacity 2?		 Capacity 2 allows the parse thread to get ahead by one batch while the embed thread is busy. This smooths out variance — if one embed call is unusually slow, parse can still work on the next batch. Capacity > 2 just accumulates memory without improving throughput (the bottleneck is parse, not the channel).
Why keep parsers
in Python?		 40+ language mappings, each with tree-sitter query files, chunk-type classifiers, and language-specific heuristics. Porting to Rust would be ~20k lines with zero performance gain — tree-sitter's C library does the heavy work. The Python parsers work from Rust callbacks as-is.
Per-file deletion
vs chunk-level diff?		 DbFileState has no chunk IDs (adding them would bloat the snapshot). For modified files: DELETE FROM chunks WHERE file_id = ? and re-insert all chunks. Tradeoff: even unchanged chunks in a modified file get re-embedded. This is correct (file changed → its embeddings should be fresh) and cheap. Chunk-level diffing (preserving old embeddings for truly unchanged chunks) can be added later via a second snapshot query.
Why keep embeddings
in Python?		 IO-bound HTTP calls. httpx/openai SDKs handle auth, rate limiting, retries. Moving to Rust saves no wall-clock time but adds maintenance burden for provider-specific logic.
Rayon threads vs
ProcessPoolExecutor?		 Rayon threads share memory, avoid fork overhead (~5–10ms per worker), and don't need pickling. Tree-sitter's C library releases the GIL during parsing — threads are nearly as parallel as processes.
Error handling? Failed parse → file marked as errored, pipeline continues. Failed embed → batch skipped (chunks stored without embeddings, user can re-embed). Store failure → pipeline aborts. Orphan channel closes propagate errors naturally: if parse panics, embed thread's recv() returns RecvError and it shuts down cleanly.

These test the recovery paths — what happens when external systems fail mid-pipeline:

ScenarioTestExpected behavior
Parse callback raises on one file test_parse_error_per_file File in PipelineReport.errors. Batch continues. Remaining files parsed normally.
Embed API fails (HTTP 429 / 401) test_embed_api_failure Embed callback raises. Pipeline catches error, stores chunks without embeddings (embedding=NULL). Pipeline continues to next batch. No panic, no hang.
DB write fails (disk full) test_db_write_failure Store thread's current transaction rolls back. Previously committed batches preserved. Pipeline reports partial results.

13. Performance Benchmarks

Benchmarking is a separate gate, not a test. It compares the Rust pipeline against the Python path on real-world-scale codebases.

Harness

# Uses a real codebase (or auto-generates a synthetic one)
uv run python scripts/bench_pipeline.py --repo /path/to/repo [--runs 5]

The script indexes the same directory with both pipelines, measuring wall-clock time, peak RSS, and output DB size. Order alternates across runs (Python→Rust→Rust→Python→...) to cancel page-cache bias; only medians are reported. Before reporting timings, the script validates that both runs produced identical file/chunk/embedding counts — a mismatch aborts.

Scenarios

ScenarioRepo sizeWhat it measures
Small cold start~500 files, empty DBBaseline overhead: parse + embed + store from scratch.
Large cold start~10K files, empty DBStress test: rayon saturation, channel throughput, DuckDB bulk insert speed.
Warm re-index~10K files, nothing changedSnapshot-only path: all files skipped. Measures thread setup/teardown overhead vs Python.
Incremental update~10K files, 1 file modifiedDiff + single-file re-index cost. Dominated by parse + embed of one file.

Metrics
